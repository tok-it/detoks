# tmux + Codex + GitHub 자동화 워크플로우를 detoks 프로젝트에 적용한 방법

팀 프로젝트를 진행하다 보면, 실제 코드 구현 못지않게 중요한 것이 있다.  
바로 **작업 단위를 어떻게 끊고, 어떻게 기록하고, 어떻게 GitHub에 반영할 것인가**에 대한 운영 방식이다.

이번 detoks 프로젝트에서는 단순히 “코드를 잘 짜는 것”을 넘어서,  
**tmux 기반 멀티 pane 작업 구조**, **Codex 중심 구현 흐름**, **작업 단위별 브랜치/커밋/푸시**,  
그리고 **PR 본문 / 칸반 / 다음 작업 프롬프트 자동 생성 흐름**까지 하나의 운영 체계로 묶어 사용하고 있다.

이 글에서는 현재 프로젝트에서 실제로 사용 중인 자동화된 작업 방식을 정리하고,  
왜 이런 구조가 팀 협업에 유리한지 소개해보려고 한다.

---

## 1. 왜 이런 자동화가 필요했나

LLM 기반 개발 도구를 쓰면 코드는 빨리 만들 수 있다.  
하지만 실제 팀 프로젝트에서는 그보다 더 자주 생기는 문제가 있다.

- 작업 범위가 섞이는 문제
- 브랜치가 뒤섞이는 문제
- 구현은 끝났는데 커밋/푸시 범위를 정리하지 못하는 문제
- PR 본문 작성이 늦어지면서 변경 이유가 흐려지는 문제
- 다음 작업으로 넘어갈 때 문맥이 단절되는 문제

즉,  
**“코드를 만드는 속도”보다 “작업 단위를 안전하게 마무리하는 방식”이 더 중요해지는 순간**이 생긴다.

detoks에서는 이 문제를 해결하기 위해,  
코드 생성과 작업 마감 흐름을 분리하고,  
각 단계에서 LLM이 맡을 역할을 고정하는 방식으로 운영을 정리했다.

---

## 2. 현재 사용하는 작업 구조

기본 구조는 tmux의 여러 pane을 역할별로 나누는 방식이다.

### 왼쪽 pane
- 하네스 엔지니어링 프롬프트 설계
- 현재 작업 단위 정의
- 다음 단계 작업 프롬프트 준비

### 오른쪽 상단 pane
- **Codex 기반 코드 구현**
- 실제 파일 수정
- 테스트 코드 추가
- 구조 변경

### 현재 pane
- 작업 단위 완료 후 범위 확인
- 브랜치 적절성 확인
- 검증 실행
- 커밋 / 푸시
- PR 본문 초안 정리

즉, 단순히 “왼쪽은 계획, 오른쪽은 구현”이 아니라,

> **구현 pane와 마감 pane의 역할을 분리하는 것**

이 핵심이다.

---

## 3. 자동화된 흐름은 어떻게 돌아가나

현재 detoks의 작업 자동화 흐름은 대략 아래 순서로 돌아간다.

### (1) 작업 시작 전
- 현재 작업이 새 브랜치가 필요한지 먼저 판단
- 필요하면 새 브랜치 생성
- 작업 범위에 맞는 하네스 프롬프트 생성

### (2) 구현 단계
- 오른쪽 Codex pane에서 코드 생성
- 관련 파일만 읽고 구현
- 필요한 최소 테스트 수행

### (3) 작업 완료 단계
- 현재 pane에서 완료 처리 프롬프트 실행
- 브랜치가 적절한지 재확인
- 관련 파일만 선택해서 커밋
- 원격 브랜치에 푸시

### (4) 푸시 이후
자동으로 다음 정보까지 이어서 정리한다.

1. PR 본문 초안  
2. 칸반 In Progress 내용  
3. 다음 작업 하네스 프롬프트  
4. PR 본문 검토 체크리스트

이렇게 되면 작업 단위 하나가 끝날 때마다  
**“구현 → 정리 → 기록 → 다음 작업 준비”**가 끊기지 않고 이어진다.

---

## 4. 왜 pane을 나눠서 쓰는가

처음에는 “오른쪽 pane에서 구현까지 했으면 거기서 바로 커밋/푸시도 하면 되는 것 아닌가?”라는 생각을 할 수 있다.  
하지만 실제로는 그 방식이 자주 문제를 만든다.

예를 들어:

- 다음 작업 초안이 이미 조금 섞여 있는 경우
- `.gitignore` 같은 로컬 운영 파일이 같이 수정된 경우
- 문서와 코드 변경이 한 브랜치에 같이 섞인 경우
- 현재 브랜치가 이미 이전 작업용인데 그대로 이어서 작업한 경우

이런 상황에서는 구현 pane가 아니라,

> **현재 pane에서 “이번 작업 단위만” 다시 확인하는 과정**

이 매우 중요해진다.

즉,

- **오른쪽 pane** = Builder
- **현재 pane** = Closer / Gatekeeper

로 역할을 나누면,  
코드 생성 흐름과 GitHub 반영 흐름이 충돌하지 않는다.

---

## 5. 현재 자동화로 얻는 장점

### 1) 작업 단위가 더 명확해진다
어떤 파일이 이번 작업에 포함되는지,  
어떤 파일은 제외해야 하는지를 항상 다시 확인하게 되므로  
커밋이 훨씬 깨끗해진다.

### 2) 브랜치 전략이 안정적이다
새 작업으로 넘어갈 때마다  
현재 브랜치가 적절한지 먼저 확인하고,  
필요하면 새 브랜치로 분리하게 된다.

즉,  
**“작업 단위”와 “브랜치 단위”가 일치하기 쉬워진다.**

### 3) PR 본문 품질이 좋아진다
푸시 직후 PR 본문 초안이 자동으로 따라오므로  
변경 이유, 영향 범위, 테스트 방법을 놓칠 확률이 줄어든다.

### 4) 다음 작업으로 넘어가는 속도가 빨라진다
작업이 끝날 때마다 다음 하네스 프롬프트가 바로 생성되기 때문에  
“다음엔 뭘 해야 하지?”를 다시 길게 정리할 필요가 없다.

### 5) 팀원이 봐도 흐름이 이해된다
PR 본문, 칸반 카드, 커밋 메시지, 브랜치명이  
하나의 작업 단위를 중심으로 정리되기 때문에  
팀원이 나중에 봐도 이해하기 쉽다.

---

## 6. 토큰 절감 측면에서도 유리하다

이 구조는 협업 안정성뿐 아니라,  
LLM 사용 비용과 문맥 효율 측면에서도 이점이 있다.

예를 들어 반복되는 공통 문맥은  
`.prompts/cli-task-context.md` 같은 파일로 고정해두고,

매번 프롬프트에 긴 설명을 복붙하는 대신:

- 공통 컨텍스트 파일 1개
- 현재 작업 관련 TS 파일 경로 몇 개
- 이번 작업 목표

만 전달하도록 바꾸면,  
프롬프트 길이가 줄고 모델이 현재 작업에 더 집중하기 쉬워진다.

즉,

> **반복되는 설명은 파일로 고정하고,  
> 매번 바뀌는 내용만 프롬프트로 주는 방식**

이 토큰 효율과 작업 정확도를 동시에 높여준다.

---

## 7. 이 방식이 특히 잘 맞는 프로젝트

이런 자동화 구조는 특히 아래 같은 프로젝트에서 효과가 크다.

- 브랜치를 자주 나눠야 하는 프로젝트
- 코드 / 문서 / PR 정리가 같이 움직이는 프로젝트
- LLM을 활용한 pair-programming 형태의 개발
- 역할이 나뉜 팀 협업
- 세션이 길어질수록 문맥 비용이 커지는 작업

detoks처럼  
CLI wrapper, adapter, subprocess, pipeline, state 경계가 있는 구조에서는  
이런 운영 자동화가 특히 잘 맞는다.

---

## 8. 느낀 점

이번에 정리한 방식의 핵심은  
“LLM이 대신 코드를 짜준다”가 아니라,

> **LLM이 팀 개발의 작업 단위를 정리하고 마감하는 데에도 참여하게 만든 것**

에 있다.

즉, 코드 생성 자체보다도

- 언제 새 브랜치를 써야 하는지
- 어떤 파일만 커밋해야 하는지
- PR 본문을 어떻게 설명해야 하는지
- 다음 작업은 무엇인지

를 자동 흐름 안에 넣은 것이 더 큰 효과를 냈다.

개인적으로는 이 방식이  
LLM을 “코드 생성기”가 아니라  
**협업 프로세스 보조 도구**로도 확장하는 좋은 예라고 느꼈다.

---

## 마무리

detoks 프로젝트에서는 현재 다음과 같은 자동화 흐름을 사용하고 있다.

- 하네스 프롬프트 생성
- 작업 단위별 브랜치 판단
- 구현 pane / 마감 pane 역할 분리
- 커밋 / 푸시 자동 보조
- PR 본문 초안 생성
- 칸반 내용 생성
- 다음 작업 프롬프트 자동 생성

이 구조 덕분에  
코드 구현뿐 아니라 **작업 단위의 시작과 끝**까지 더 깔끔하게 정리할 수 있게 되었다.

앞으로는 여기에

- 실제 runtime 실행 정책
- 브랜치 전환 자동화
- PR 생성 자동화

까지 더 붙이면,  
LLM 기반 팀 개발 워크플로우를 더 완성도 있게 가져갈 수 있을 것 같다.
