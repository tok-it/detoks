# 모델 역할 분리와 task 분류 전략에서 느낀 점

## 한 줄 요약

LLM을 어디에 쓰고 어디서 멈춰야 하는지 고민하면서, “모델이 잘하는 일”과 “코드가 잘하는 일”을 구분하는 감각이 중요하다는 걸 느꼈다.

## 개발 배경

이번에 가장 고민했던 부분은 모델을 많이 쓰는 게 좋은지, 아니면 아껴 써야 하는지였다.

결론적으로는 모든 걸 모델에 맡기는 것보다,
의미 재구성이 필요한 부분만 모델에 맡기고
닫힌 집합의 분류는 코드가 처리하는 쪽이 더 안정적이었다.

## 개발 과정에서 얻은 인사이트

### 1) 모델은 의미를 다듬는 데 강하다

한국어를 영어로 압축하거나,
장황한 결과를 요약하는 일은 단순 문자열 처리가 아니라
의도를 보존하면서 표현을 다시 만드는 작업이다.

이런 영역은 여전히 모델의 장점이 분명했다.

### 2) 분류는 생각보다 코드가 더 잘 맞는다

처음에는 분류도 모델이 알아서 해주면 편할 것 같았지만,
실제로는 create / analyze / modify / validate / document 같은 닫힌 집합에서는
규칙 기반 분류가 더 예측 가능했다.

특히 디버깅할 때
“왜 이 요청이 이 카테고리로 갔는지”
를 설명할 수 있다는 점이 컸다.

### 3) 하나의 모델을 쓰는 것과 하나의 단계로 묶는 것은 다르다

Prompt Compiler, Request Analyzer, Context Summarizer는
같은 모델을 사용할 수는 있어도,
출력 형식과 실패 포인트가 다르기 때문에 분리해야 했다.

이걸 한 번에 묶으면 편해 보이지만,
문제 발생 시 원인 분리가 어려워진다.

### 4) 애매한 입력은 예외 처리의 가치가 크다

모든 요청이 깔끔하게 분류되지는 않는다.
그래서 기본은 코드 분류로 가고,
정말 애매할 때만 보조적인 방식으로 넘어가는 구조가 더 현실적이라고 느꼈다.

## 배운 점

- LLM은 전천후 도구가 아니라 강점이 있는 구간에 집중시킬 때 좋다.
- 분류처럼 닫힌 문제는 코드가 더 안정적이다.
- 단계별 분리는 품질보다도 디버깅과 유지보수에 큰 영향을 준다.
- 예외 케이스를 위한 fallback 구조가 전체 시스템을 더 단단하게 만든다.

## 느낀 점

이번 정리를 하면서, 모델 중심 설계를 하더라도 결국은 “경계 설계”가 더 중요하다는 걸 느꼈다.
모델이 똑똑하더라도 역할을 흐리게 두면 시스템 전체가 애매해진다.

반대로 역할을 잘 나눠두면 모델은 필요한 순간에만 쓰이고,
코드는 예측 가능하게 동작한다.

그 균형을 잡는 게 생각보다 중요한 작업이었다.

## 정리한 방향

1. 입력 수신
2. 한국어 → 영어 압축
3. 요청 분류
4. task 추출
5. context 요약
6. 상태 저장
