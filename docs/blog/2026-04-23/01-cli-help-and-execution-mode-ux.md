# detoks CLI 도움말과 실행 모드 UX에서 배운 점

## 한 줄 요약

CLI는 기능이 동작하는 것보다, 사용자가 지금 무엇을 해야 하는지 즉시 이해하게 만드는 것이 더 중요하다는 점을 다시 느꼈다.

## 개발 배경

이번 작업을 하면서 가장 먼저 보였던 문제는 “기능 자체”가 아니라 “길 안내”였다.

- 도움말 진입점이 분명하지 않았고
- 잘못된 입력을 넣었을 때 사용자가 다음 행동을 알기 어려웠고
- REPL이 어떤 실행 모드로 시작됐는지 눈에 잘 들어오지 않았다.

처음엔 사소한 UX처럼 보였지만, 실제로는 CLI 신뢰도에 직접 영향을 주는 부분이었다.

## 개발 과정에서 얻은 인사이트

### 1) 에러 메시지는 실패 설명이 아니라 다음 행동 안내여야 한다

단순히 `unknown flag`라고만 말하면 사용자는 멈춘다.
이번에 메시지에 `detoks --help` 같은 안내를 넣으면서,
에러는 끝점이 아니라 재진입점이 되어야 한다는 걸 다시 확인했다.

### 2) help는 옵션이 아니라 기본 안전장치다

`--help`와 `-h`를 따로 지원한 이유는 편의성보다 심리적 안정감에 가깝다.
사용자는 언제든 “내가 뭘 할 수 있는지” 확인할 수 있어야 한다.

### 3) 실행 모드는 로그가 아니라 맥락이다

REPL 시작 시 `adapter`만 보이면 반쪽짜리 정보다.
`executionMode`까지 함께 보여줘야 지금 이 세션이 stub인지 real인지 바로 판단할 수 있다.

### 4) 파서는 작은 변화에도 쉽게 깨진다

help, flag 우선순위, 오류 메시지처럼 사소해 보이는 변화도 파서에서는 회귀가 발생하기 쉽다.
그래서 단위 테스트로 고정해 두는 게 중요했다.

## 배운 점

- CLI UX는 문서보다 먼저 “즉시성”이 중요하다.
- 도움말은 부가 기능이 아니라 사용자 보호 장치다.
- 오류 메시지는 원인 설명보다 복구 경로를 같이 주는 편이 낫다.
- 파서 로직은 테스트를 붙여야 다음 옵션 추가 때 안전하다.

## 느낀 점

이번 수정은 코드 양 자체는 크지 않았지만, 체감 품질은 꽤 크게 바뀌는 작업이었다.
기능 추가보다 이런 UX 정리가 사용자 경험에 더 큰 영향을 줄 수 있다는 걸 실감했다.

특히 CLI는 한 번 막히면 사용자가 바로 떠나기 쉬워서,
작은 안내 하나가 사용성을 좌우한다는 점이 인상적이었다.
