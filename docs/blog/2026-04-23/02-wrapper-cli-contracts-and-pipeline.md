# wrapper CLI 계약 정리에서 배운 점

## 한 줄 요약

CLI를 “실행 코드”로만 보면 복잡도가 금방 섞이고, 오케스트레이션 계층으로 분리해서 바라봐야 구조가 흔들리지 않는다는 걸 배웠다.

## 개발 배경

프로젝트가 커질수록 가장 먼저 무너지는 건 기능이 아니라 경계였다.

- CLI가 어디까지 책임지는지
- core와 integrations의 차이가 무엇인지
- 단계별 입력/출력이 어떤 계약을 따르는지

이게 흐려지면 기능은 돌아가도 유지보수가 어려워진다.

그래서 이번에는 구현보다 먼저 “누가 무엇을 소유하는가”를 문서로 정리했다.

## 개발 과정에서 얻은 인사이트

### 1) 계약이 먼저 있어야 구현이 흔들리지 않는다

`docs/API_SPEC.md`를 정리하면서 느낀 점은,
내부 API라도 계약이 없으면 결국 구현이 임시방편으로 흐르기 쉽다는 것이다.

스키마, envelope, boundary validation이 있는 순간부터
비로소 각 단계의 책임이 보이기 시작했다.

### 2) wrapper CLI는 똑똑할수록 좋은 게 아니다

CLI가 모든 판단을 직접 처리하면 편해 보이지만,
실제로는 역할이 커질수록 재사용성과 테스트성이 떨어진다.

이번에 wrapper CLI를 얇은 오케스트레이션 계층으로 정의하면서,
“CLI는 연결하고, 판단은 적절한 레이어에 남긴다”는 원칙이 더 낫다고 느꼈다.

### 3) 단계 순서를 고정하면 논쟁이 줄어든다

입력 → 파싱 → 세션/모드 → 프롬프트 정제 → 요청 분석 → task 분해 →
문맥 최적화 → 실행 → 출력 처리 → 상태 저장

이 순서를 정해두면 구현 논의가 추상적이지 않고,
어느 단계가 책임을 가져야 하는지 훨씬 명확해진다.

### 4) 문서는 나중에 쓰는 설명이 아니라 설계 결정 기록이다

이번 문서는 기능 설명서처럼 보일 수 있지만,
실제로는 앞으로 코드가 어디까지 확장되어도 되는지 정하는 기준선 역할을 한다.

## 배운 점

- 내부 API도 계약 중심으로 봐야 한다.
- CLI는 기능의 중심이 아니라 흐름의 시작점이다.
- 경계를 명확히 해야 리팩터링이 쉬워진다.
- 문서화는 구현 후 작업이 아니라 설계 과정의 일부다.

## 느낀 점

이 작업을 하면서 “정리”의 가치가 꽤 크다는 걸 느꼈다.
코드를 많이 바꾸지 않아도, 책임의 경계를 문서로 적어두는 것만으로
앞으로의 작업 속도와 안정감이 달라질 수 있다.

특히 혼자 개발할 때도 이런 경계 정리는 나중에 스스로를 돕는 메모가 된다.
