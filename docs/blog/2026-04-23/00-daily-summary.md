# 2026-04-23 개발 일지

## 오늘의 목표

오늘의 목표는 크게 세 가지였다.

1. CLI 사용성을 높이기 위한 도움말/오류 메시지 개선
2. wrapper CLI와 내부 파이프라인의 책임 경계 정리
3. 모델 역할과 task 분류 전략에 대한 기준 정리

단순히 기능을 추가하는 것보다, 앞으로 작업할 때 흔들리지 않을 기준을 문서와 코드에 남기는 데 초점을 맞췄다.

## 내일의 계획

내일은 오늘 정리한 내용을 바탕으로 다음을 진행하고 싶다.

- help 문구를 command 단위로 더 세분화
- wrapper CLI 관련 문서와 실제 구현 간 용어 일치 점검
- task 분류 규칙과 fallback 기준 보완
- 필요하다면 오늘 작성한 md를 블로그 게시용 문체로 추가 다듬기

## 실제 수행한 작업

### 1) CLI 도움말과 실행 모드 UX 정리

- `--help`, `-h` 지원을 추가했다.
- 잘못된 입력에 대한 오류 메시지에 사용법 안내를 포함했다.
- REPL 시작 시 `adapter`뿐 아니라 `executionMode`도 함께 표시하도록 정리했다.
- 파서 동작을 단위 테스트로 고정했다.

이 작업을 통해 CLI는 단순히 명령을 받는 도구가 아니라,
사용자가 길을 잃지 않도록 안내하는 도구라는 점을 다시 느꼈다.

### 2) wrapper CLI 계약과 파이프라인 책임 분리 정리

- 내부 API 계약 문서를 정리했다.
- wrapper CLI가 무엇을 책임지고, 무엇을 넘겨야 하는지 문서화했다.
- 입력 → 파싱 → 세션/모드 → 프롬프트 정제 → 요청 분석 → task 분해 → 실행 → 출력 처리 → 상태 저장의 흐름을 정리했다.

이 작업은 구현보다 먼저 경계를 정하는 일에 가까웠고,
나중에 코드가 커져도 흔들리지 않을 기준선을 만드는 작업이었다.

### 3) 모델 역할 분리와 task 분류 전략 정리

- LLM이 맡아야 할 영역과 코드가 맡아야 할 영역을 다시 나눴다.
- 한국어 → 영어 변환, 출력 압축, 상태 요약은 모델 쪽에 두는 것이 적합하다고 판단했다.
- 요청 분류와 task 명확화는 닫힌 집합 문제이므로 코드 레벨 분류가 더 안정적이라는 방향을 정리했다.

이 작업을 하면서 모델을 “많이 쓰는 것”보다 “적절한 곳에 쓰는 것”이 더 중요하다는 점을 확인했다.

## 이슈 상황 및 해결방안

### 이슈 1. 도움말 부재로 인한 진입 장벽

**상황**
- 사용자가 옵션을 모를 때 어디서 시작해야 할지 불명확했다.
- 잘못된 플래그 입력 시 에러만 표시되고 끝날 수 있었다.

**해결방안**
- `--help`, `-h`를 명시적으로 지원했다.
- 에러 메시지에 `detoks --help` 같은 다음 행동을 함께 안내했다.

### 이슈 2. wrapper CLI 책임이 넓어질 위험

**상황**
- CLI가 모든 결정을 직접 처리하면 구조가 금방 섞일 수 있었다.
- 역할 경계가 흐려지면 유지보수가 어려워질 수 있었다.

**해결방안**
- CLI를 얇은 오케스트레이션 계층으로 정의했다.
- 내부 API와 단계별 책임을 문서로 고정했다.

### 이슈 3. 모델과 코드의 책임 경계가 애매해질 수 있음

**상황**
- 분류까지 모델에 맡기면 편해 보이지만, 비용과 일관성 측면에서 불리할 수 있었다.
- 반대로 모든 걸 코드로 처리하면 의미 재구성이 필요한 작업이 약해질 수 있었다.

**해결방안**
- 의미 압축은 모델, 닫힌 집합 분류는 코드라는 원칙으로 정리했다.
- 애매한 요청만 예외적으로 보조 경로를 고려하는 구조로 방향을 잡았다.

## 느낀점

오늘 작업은 기능 추가보다 “정리”와 “경계 설정”이 더 많은 비중을 차지했다.
처음에는 문서화나 메시지 개선이 부가적인 일처럼 느껴질 수 있지만,
실제로는 프로젝트의 유지보수성과 사용성을 크게 좌우하는 핵심 작업이라는 걸 느꼈다.

특히 CLI와 모델 관련 작업은 한 번 구조가 흐려지면 나중에 다시 정리하기가 어렵기 때문에,
오늘처럼 기준을 문서와 코드에 같이 남겨두는 방식이 꽤 유효하다고 느꼈다.

## 오늘의 정리

- 사용성은 기능만으로 완성되지 않는다.
- 경계가 명확해야 코드가 커져도 흔들리지 않는다.
- 모델은 모든 것을 맡기는 대상이 아니라, 잘하는 영역에 집중시켜야 한다.
- 문서화는 나중 작업이 아니라 설계의 일부다.
