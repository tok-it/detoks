# 후속 단계 전달 텍스트 크기 절감의 기준과 계산 근거

> detoks 관점 정리 문서  
> 핵심 질문: **이미 생성된 원본 출력 자체**가 아니라, **그 이후 단계로 넘기는 텍스트 크기**를 어떻게 절감으로 볼 것인가?

---

## 1. 이 문서가 다루는 범위

detoks에서 “토큰 절감”은 하나로 뭉뚱그리면 오해가 생깁니다.

왜냐하면 다음 3가지는 서로 다른 비용이기 때문입니다.

1. **실제 생성 비용**
   - LLM이 답변을 생성할 때 이미 소모된 토큰
2. **후속 단계 전달 비용**
   - 생성된 결과를 다음 LLM 호출이나 다음 파이프라인 단계에 넘길 때 드는 토큰
3. **저장 / 표시 비용**
   - 세션 파일, 로그, 대시보드, REPL 화면에 쌓이는 텍스트 크기

이 문서는 그중에서도 **2번, 후속 단계 전달 비용 절감**에 초점을 둡니다.

---

## 2. 먼저 결론

detoks 관점에서 “절감”이라고 부를 수 있는 것은 다음입니다.

> **원본 텍스트를 다음 단계로 그대로 넘겼을 때의 토큰 수**와  
> **압축/요약/정리된 텍스트를 넘겼을 때의 토큰 수**의 차이

즉, 계산식은 단순합니다.

```text
downstream_saved_tokens = max(0, original_tokens - optimized_tokens)
downstream_saved_percent = original_tokens > 0
  ? (downstream_saved_tokens / original_tokens) * 100
  : 0
```

여기서 중요한 점은:

- **원본 출력이 이미 생성됐다면**, 그 생성 토큰은 이미 사용된 상태
- 따라서 절감으로 보는 대상은 **생성 자체**가 아니라 **그 다음 단계로 넘기는 텍스트 크기**

---

## 3. 왜 “출력 토큰 절감”이 아니라 “후속 단계 전달 텍스트 절감”인가

이 구분이 중요한 이유는 다음과 같습니다.

### 3.1 생성 비용은 되돌릴 수 없다

LLM이 긴 `raw_output`을 만들어냈다면, 그 순간 출력 토큰은 이미 소모되었습니다.

그 뒤에 `summary`를 만들었다고 해서:

- 원본 생성 토큰이 환급되지는 않고
- 이미 수행된 생성 비용이 사라지지도 않습니다

즉, **원본 출력이 이미 나온 뒤의 요약은 생성 비용 절감이 아니라 downstream 최적화**입니다.

### 3.2 다음 호출의 비용은 줄일 수 있다

하지만 그 원본 출력을 다음 단계에 그대로 다시 넣으면:

- 다음 프롬프트가 길어지고
- 컨텍스트 윈도우 압박이 커지고
- 다음 호출의 입력 토큰이 증가하고
- 응답 지연과 비용이 함께 늘어납니다

따라서 detoks가 줄이려는 것은 **“이미 써버린 출력 토큰”**이 아니라  
**“그 출력이 이후 단계로 재투입될 때 발생하는 추가 비용”**입니다.

---

## 4. 계산 순서

detoks에서 후속 단계 텍스트 절감을 계산할 때는 아래 순서가 가장 자연스럽습니다.

### 4.1 원본 텍스트와 최적화 텍스트를 정한다

예시:

- 원본 텍스트: `raw_output`, 긴 상태 설명, verbose 로그
- 최적화 텍스트: `summary`, 압축된 컨텍스트, 다음 단계용 handoff 문장

중요한 점은 최적화 텍스트가 **다음 LLM 호출이나 다음 작업 단계에 실제로 넘길 수 있는 형태**여야 한다는 것입니다.

### 4.2 같은 토크나이저로 센다

문자 수가 아니라 **토큰 수**를 비교해야 합니다.

이유:

- 한국어, 영어, 코드, JSON은 문자 수와 토큰 수의 비율이 다름
- 모델마다 토큰화 방식이 다름

그래서 detoks는 공통 비교 기준으로 **같은 토크나이저**를 사용해야 합니다.

### 4.3 차이를 계산한다

```text
savedTokens = max(0, originalTokens - optimizedTokens)
savedPercent = originalTokens > 0 ? savedTokens / originalTokens * 100 : 0
```

이 값은 **절대 절감량**과 **상대 절감률**을 함께 보여주는 것이 좋습니다.

### 4.4 해석한다

이 수치는 다음 의미를 가집니다.

- 다음 LLM 호출에 들어갈 입력이 얼마나 짧아졌는가
- 세션 상태와 로그가 얼마나 가벼워졌는가
- 재진입 시 읽어야 하는 정보량이 얼마나 줄었는가

---

## 5. detoks에서 이 절감이 실제로 유효한 이유

### 5.1 컨텍스트 윈도우가 유한하다

LLM은 무한한 문맥을 볼 수 없습니다.

따라서 텍스트를 줄이면:

- 같은 창 안에 더 많은 핵심 정보를 담을 수 있고
- 오래된 불필요 정보가 밀려나는 문제를 줄일 수 있습니다

### 5.2 입력 토큰은 비용과 지연에 직접 연결된다

후속 단계가 LLM 호출이라면, 입력이 길수록:

- 추론 비용이 증가하고
- 응답 시간이 길어질 가능성이 높고
- 실패 확률이나 문맥 누락 가능성도 커집니다

### 5.3 사람이 다시 읽을 때도 효율이 좋다

detoks는 자동화만이 아니라 **오랜만에 다시 들어온 사용자**도 봐야 합니다.

그래서 텍스트 축약은:

- 세션 목록
- 홈 대시보드
- 체크포인트 복구 화면
- REPL 요약

같은 UI에도 직접적인 이익이 있습니다.

---

## 6. detoks 관점에서의 올바른 용어

기능 이름을 정할 때는 혼동을 줄이는 것이 좋습니다.

### 권장 표현

- **후속 단계 입력 절감**
- **후속 전달 텍스트 절감**
- **다운스트림 텍스트 절감**

### 조심해서 써야 할 표현

- **출력 토큰 절감**
  - 이미 생성된 원본 출력이 있다면, 이 표현은 실제 생성 비용 감소처럼 오해될 수 있음

### 실무적으로는 이렇게 구분하면 좋다

| 항목 | 의미 |
|---|---|
| 실제 생성 토큰 | LLM이 이미 써버린 비용 |
| 후속 전달 토큰 | 다음 단계로 넘길 때 절약되는 비용 |
| 저장 토큰 | 디스크/로그/표시에서 줄어드는 비용 |

---

## 7. detoks 파이프라인에서 어디에 적용되는가

### 7.1 Role 3 작업 실행 후

작업 결과가 길게 나오면 그 결과를 그대로 다음 단계에 넣지 않고:

- 핵심만 요약하거나
- 다음 작업이 이해할 수 있는 수준으로 압축하거나
- 필요한 메타데이터만 남기는 방식으로 줄입니다

### 7.2 세션 상태 저장 시

세션 파일은 오래 남기 때문에, 장기 보관 관점에서 원본 전체를 넣는 것보다:

- 요약
- 상태 플래그
- 필요한 핵심 결과

만 남기는 편이 더 유리합니다.

### 7.3 홈 / 목록 / 재진입 UI

사람이 보는 화면에서는 원문보다:

- 마지막 작업 요약
- 다음 액션
- 절감량

이 훨씬 더 유용합니다.

---

## 8. “진짜 절감”과 “후속 절감”의 차이

이 부분은 가장 중요합니다.

### 8.1 진짜 절감

원본 답변이 애초에 짧게 생성되도록 만들었다면:

- 출력 토큰이 실제로 덜 쓰입니다
- 빌링이나 추론 비용도 줄 수 있습니다

이 경우가 **실제 생성 비용 절감**입니다.

### 8.2 후속 절감

이미 긴 답변이 나왔지만, 그 뒤에 요약해서 다음 단계에 짧게 넘겼다면:

- 원본 생성 비용은 이미 발생
- 하지만 다음 단계 입력은 줄어듭니다

이 경우가 **후속 단계 전달 텍스트 절감**입니다.

### 8.3 net benefit를 보려면 추가 비용도 봐야 한다

만약 요약 자체가 또 다른 LLM 호출로 생성된다면,
순절감은 아래처럼 보는 것이 더 정확합니다.

```text
net_benefit = downstream_saved_tokens - summary_generation_cost
```

즉, 요약을 만드는 데 든 비용보다  
그 요약 덕분에 줄어든 downstream 비용이 커야 실제로 이득입니다.

---

## 9. 예시

### 예시 1 — 원본 출력이 긴 경우

```text
원본 출력: 1,200 tokens
요약 출력:   180 tokens
절감량:     1,020 tokens
절감률:      85.0%
```

이 숫자는 “1,020토큰을 돌려받았다”는 뜻이 아닙니다.

의미는 단 하나입니다.

> **다음 단계가 이 텍스트를 다시 읽을 때, 1,020토큰만큼 덜 부담한다**

### 예시 2 — 같은 의미지만 표현만 다른 경우

어떤 요약이 원문보다 글자 수는 비슷해 보여도,
토큰화 결과가 다를 수 있습니다.

특히:

- 코드
- JSON
- 한국어
- 영어 혼합 문장

은 문자 수와 토큰 수가 일치하지 않기 때문에,
**반드시 토큰 기준으로 비교**해야 합니다.

---

## 10. detoks에서의 권장 해석

detoks는 다음과 같이 해석하는 것이 가장 안전합니다.

1. **원본 출력이 생성되면 생성 비용은 이미 발생한 것**
2. **요약/압축은 그 다음 단계의 텍스트 크기를 줄이는 수단**
3. **토큰 절감 수치는 downstream efficiency 지표로 본다**
4. **billing 절감과 downstream 절감은 같은 값이 아니다**

---

## 11. 실무 결론

detoks 관점에서 후속 단계 텍스트 절감은 매우 유용한 지표입니다.

다만 이 지표를 다음처럼 해석해야 혼동이 없습니다.

- **생성 자체를 줄인 절감**: 실제 비용 절감
- **후속 단계 전달을 줄인 절감**: downstream 비용 절감

따라서 UI와 문서에서는 가능하면:

> “출력 토큰 절감”보다  
> “후속 단계 전달 텍스트 절감” 또는 “다운스트림 텍스트 절감”

이라는 표현을 쓰는 편이 더 정확합니다.

---

## 12. 한 줄 요약

**원본 출력이 이미 생성된 뒤라면 그 토큰은 이미 사용된 것이고, detoks가 절감으로 볼 수 있는 것은 그 결과를 다음 단계에 넘길 때 줄어드는 텍스트 크기다.**
