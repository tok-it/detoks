> 하나의 모델이 여러 지능형 전처리/압축 작업을 수행하되, 결과는 역할별 파이프라인 단계로 분리해 다룬다.

  즉,

  - 모델은 하나일 수 있음
  - 하지만 기능은 논리적으로 분리해야 합니다.

  ———

  # 결론

  당신이 말한 3가지:

  1. 프롬프트를 영어로 변환하면서 문맥 유지 + 압축
  2. 개발 키워드/요청 카테고리 분류
  3. 세션의 작업 문맥 압축

  이 세 가지를 하나의 LLM이 담당하는 방향은 충분히 가능하고,
  오히려 MVP 단계에서는 가장 현실적인 접근입니다.

  하지만 중요한 건:

  > “모델 하나”와 “기능 하나”는 다르다

  입니다.

  즉 하나의 모델을 써도 내부적으로는 아래처럼 분리해야 합니다.

  - Prompt Compiler
  - Request Analyzer
  - Context Summarizer

  ———

  # 왜 하나의 모델로 가능하냐

  이 세 작업은 공통적으로 전부 언어 이해 + 재구성 문제입니다.

   ## 공통점

  - 의미 유지
  - 정보 선택
  - 불필요한 표현 제거
  - 구조화된 결과 생성

  즉 전부 LLM이 잘하는 영역입니다.

  그래서 초기에는:

  - 별도 모델 여러 개
  - 별도 ML classifier
  - 별도 압축 엔진

  이렇게 나누는 것보다

  하나의 강한 모델에 역할별 프롬프트를 주는 방식이 더 단순하고 빠릅니다.

  ———

  # 하지만 왜 논리적으로는 분리해야 하냐

  이유는 세 가지입니다.

  ## 1) 출력 형식이 다름

  ### Prompt Compiler 출력

  예:

  {
    "compressed_prompt": "Implement a TypeScript CLI command parser..."
  }

  ### Request Analyzer 출력

  예:

  {
    "category": "feature_implementation",
    "keywords": ["typescript", "cli", "parser"],
    "tasks": [...]
  }

  ### Context Compression 출력

  예:

  {
    "summary": "...",
    "active_files": [...],
    "next_action": "..."
  }

  즉 결과물이 다르기 때문에
  하나의 모델을 쓰더라도 단계를 분리해야 합니다.

  ———

  ## 2) 실패 지점이 다름

  - 영어 압축은 잘 됐는데 분류가 틀릴 수 있음
  - 분류는 맞았는데 세션 요약이 손실될 수 있음
  - 세션 압축은 됐는데 키워드 추출이 약할 수 있음

  기능을 한 덩어리로 합치면
  어디서 잘못됐는지 추적이 어려워집니다.

  ———

  ## 3) 나중에 교체 가능해야 함

  초기엔 하나의 모델로 다 처리해도,
  나중에는 이런 식으로 바뀔 수 있습니다.

  - Prompt Compiler → Python Role1
  - Category Classifier → rule-based + LLM hybrid
  - Context Compression → 별도 summarization path

  즉 아키텍처는 처음부터 모듈형이어야 합니다.

  ———

  # 그래서 이 프로젝트 방향성을 정확히 말하면

  ## 맞는 표현

  > 하나의 LLM을 중심으로
  > 프롬프트 압축, 요청 분류, 세션 문맥 압축을 수행하되,
  > 이를 각각 독립된 파이프라인 단계로 분리하여 관리하는 구조

  이게 가장 정확합니다.

  ———

  # 추천 처리 구조

  ## 입력

  사용자 요청:

  > "codex-cli에서 plus 구독 기준 사용할수있는 모델..."

  ## Step 1. Prompt Compiler

  - 한국어 → 영어
  - 의미 유지
  - 토큰 절감
  - 장황한 표현 제거

  ## Step 2. Request Analyzer

  - 질문 유형 분류
  - 개발 키워드 추출
  - task 단위 추출

  ## Step 3. Context Optimizer / Session Compressor

  - 현재 작업과 관련 있는 기존 세션만 압축
  - 다음 턴에 필요한 정보만 남김

  이 3단계를 동일 모델이 수행할 수 있음

  ———

  # 하나의 호출로도 가능하냐?

  ## 가능은 합니다

  예를 들어 모델에 한 번에:

  - compressed prompt
  - category
  - keywords
  - summary

  를 모두 JSON으로 뽑게 할 수는 있습니다.

  예:

  {
    "compressed_prompt": "...",
    "category": "bug_fix",
    "keywords": ["typescript", "cli"],
    "session_summary": "...",
    "next_action": "..."
  }

  ———

  ## 하지만 초기 설계에서는 추천하지 않음

  왜냐하면 너무 많은 책임을 한 번에 묶으면:

  - 디버깅 어려움
  - 품질 튜닝 어려움
  - 실패 시 전체 재호출 필요
  - 부분 재사용 어려움

  ———

  # 더 좋은 방향

  ## 같은 모델, 다른 단계

  즉 이렇게:

  - 모델은 하나
  - 프롬프트 템플릿은 단계별로 다름
  - 출력 스키마도 단계별로 다름

  이게 가장 좋습니다.

  ———

  # 기술적으로 요약하면

  ## 이 프로젝트의 방향성은

  - 여러 문제를 전부 하나의 모델이 처리하는 것은 맞을 수 있음
  - 하지만 그걸 한 번의 거대한 작업으로 보는 건 위험함
  - 하나의 모델을 여러 파이프라인 단계에서 재사용하는 구조가 적절함

  ———

  # 한 줄 답

  > 네, 초기 방향성으로는 하나의 모델이 영어 압축, 개발 카테고리 분류, 세션 문맥 압축을 모두 처리하는 것이 맞습니다. 다만 그것을 한 덩어리 기능으로 합치기보다, 하나의 모델을 단계
  > 별 파이프라인에서 재사용하는 구조로 설계하는 것이 이 프로젝트에 더 적합합니다.
