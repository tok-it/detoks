왜 코드 레벨이 1순위인가

네 상황을 다시 보면 분류 대상은:

create
analyze
modify
validate
document

같은 닫힌 집합의 대분류입니다.

그리고 입력도:

개발 요청
비교적 짧은 프롬프트
반복적으로 등장하는 표현

이런 성격을 갖습니다.

이 경우 핵심은:

빠르게
일관되게
싸게
예측 가능하게

분류하는 것입니다.

이 기준에서는 코드 레벨 분류가 훨씬 유리합니다.

1. 코드 레벨 분류 방식

예를 들면 이런 구조입니다.

방식
대분류별 동의어 사전 정의
입력을 normalize
토큰화
각 카테고리와 점수 계산
가장 높은 점수 선택

예:

create: make, generate, build, create, implement
analyze: explain, inspect, analyze, why, reason
modify: fix, refactor, change, improve, edit
validate: test, verify, check, validate
document: document, comment, write readme, explain usage

장점
1) 빠름

LLM 호출도 없고, 벡터 검색 인프라도 없어도 됩니다.

2) 일관성 높음

같은 입력이면 같은 결과가 나옵니다.

3) 디버깅 쉬움

왜 modify로 갔는지 설명 가능합니다.

4) 구현 단순

MVP에 매우 적합합니다.

5) 비용 거의 없음

토큰도, 외부 검색도 필요 없습니다.

단점
1) 표현 다양성에 약함

동의어 사전에 없는 표현이 나오면 놓칠 수 있습니다.

2) 문맥 해석이 약함

예:

“테스트를 고려해서 구현해줘”
이걸 create로 볼지 validate로 볼지 애매할 수 있습니다.
3) 복합 요청 처리 한계

한 문장에 여러 task가 섞이면 정확도가 떨어질 수 있습니다.

2. RAG + 코사인 유사도 방식

이건 보통 이렇게 됩니다.

방식
각 task category에 대한 설명/예시 문장들을 문서처럼 저장
사용자 입력을 임베딩
category 예시들과 코사인 유사도 계산
가장 가까운 카테고리 선택

예:

create 예시 문장 20개
analyze 예시 문장 20개
modify 예시 문장 20개
...

그리고 입력:

“이 에러 원인을 찾고 수정 방향까지 알려줘”

를 벡터화해서 유사도 비교

장점
1) 표현 다양성에 강함

동의어 사전에 없는 표현도 의미적으로 가까우면 잡을 수 있습니다.

2) 확장성 좋음

카테고리별 예시를 추가하면서 성능을 올릴 수 있습니다.

3) 문장 수준 유사성 반영 가능

단어 일치보다 의미 일치에 강합니다.

단점
1) “RAG”라고 부르기엔 과한 경우가 많음

이건 사실 엄밀히 말하면 RAG라기보다 벡터 기반 분류/검색에 더 가깝습니다.

2) 인프라 복잡도 증가
embedding 모델
벡터 저장소
retrieval logic
가 필요할 수 있습니다.
3) 닫힌 소분류 문제에는 과할 수 있음

분류 라벨이 5개 정도로 고정되어 있다면 너무 무거운 해법일 수 있습니다.

4) 디버깅이 어려움

왜 create보다 modify가 더 가깝게 나왔는지 설명이 애매할 수 있습니다.

5) 실시간 CLI 분류에서 오버헤드 가능

detoks는 인터랙티브 CLI이므로, 첫 단계 분류는 가벼워야 합니다.

3. RAG라는 표현이 정확한가

여기서 하나 짚고 넘어가면 좋습니다.

네가 말한 방식은 엄밀히는:

RAG라기보다 embedding-based retrieval classifier

에 더 가깝습니다.

RAG는 보통:

검색된 문서를
생성 모델 입력에 붙여
생성 품질을 높이는 구조

를 말합니다.

그런데 지금 문제는 “분류”입니다.
즉 생성보다 routing/classification 문제입니다.

그래서 이 단계에선:

RAG
보다는
vector similarity classification
이라고 보는 게 더 정확합니다.

4. 어떤 방법이 더 맞는가
MVP 기준

코드 레벨이 맞습니다.

이유:

카테고리가 적음
규칙이 비교적 명확함
빠르고 싸고 디버깅 쉬움
CLI 프로젝트에 적합
고도화 단계

다음 상황이면 벡터 기반 보조 분류를 붙일 가치가 있습니다.

사용자 표현 다양성이 커짐
오분류가 자주 발생함
카테고리 수가 늘어남
동의어 사전 유지보수가 어려워짐
영어/한국어/혼합 표현이 많이 들어옴

즉,

처음엔 코드 레벨, 필요할 때 벡터 보조

가 가장 현실적입니다.

5. 가장 추천하는 구조
하이브리드 방식
1단계: 코드 레벨 1차 분류
동의어 사전
키워드 매칭
점수 계산
2단계: 애매할 때만 벡터 보조

예:

최고 점수가 낮음
1, 2위 점수 차이가 작음
복합 요청 가능성 높음

그때만:

example embedding lookup
cosine similarity 재판단
3단계: 정말 애매하면 모델 보조
task extraction prompt
structured output

이 3단계 구조가 가장 좋습니다.
